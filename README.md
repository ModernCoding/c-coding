To get a clear, data-driven answer about why one version of your code is faster than the other, you should use a profiler.

A profiler is a tool that monitors your program as it runs, counting exactly how many times each function is called and how many CPU cycles are spent on each specific line of code.

Recommended Tools
gprof (GNU Profiler): The classic tool for C programs. It tells you exactly where your time is going.

valgrind (specifically the callgrind tool): This is incredibly powerful. It simulates the CPU and cache to give you a "cost" for every single instruction, which would show you exactly how much time is spent on your pointer dereferences versus loop logic.

How to use gprof (Quick Guide)
Compile with profiling flags: Add the -pg flag to your GCC command:
gcc -pg your_code.c -o fib_test

Run your program: Just execute it normally: ./fib_test

This will create a file called gmon.out.

Generate the report: Run gprof fib_test gmon.out > analysis.txt

Read the report: The analysis.txt file will show you a "flat profile," which lists your functions and the percentage of time the CPU spent inside each one.

Why this is a "Game Changer" for understanding
When you look at the raw timing data, you might be surprised. Often, recursive functions seem faster because they take a different path through the CPU's branch predictor, but they might actually be using more memory bandwidth or causing more cache misses than a simple loop. A profiler removes the guesswork and shows you the "truth" of what the hardware is doing.

A final tip: If you want to see if the compiler is the reason for the speed difference, try compiling with -S. This will output the assembly code generated by the compiler. Comparing the assembly code for your fibonacciLoop and fibonacciRecursive will show you exactly how the compiler translated your C code into machine instructions—you will likely see fewer "jump" instructions in the version that the compiler favored.
